# WarpParse 0.24

Dear users,

We are pleased to announce the release of WarpParse 0.24. This update focuses on **HTTP ingestion, remote project layout, error-governance improvements, and fixes around OML / SQL parsing details**.

## Highlights

- Added HTTP Source to complete the HTTP push-based ingestion path.
- Added dual-repository support for remote projects, allowing `models/` and `infra/` to be synced separately.
- Upgraded the error-governance stack to make CLI failures more complete and easier to read.
- Fixed OML `take(...)` behavior and improved SQL parameter parsing stability.

## Added

### HTTP Source

Added a general-purpose `HTTP Source` for receiving externally pushed data over HTTP.

Current capabilities include:

- `json` and `ndjson` input formats
- Coordinated format selection through `Content-Type` and the `fmt` query parameter
- `gzip`-compressed request bodies
- Ingestion routing based on `port` + `path`

Related docs:

- [HTTP Source configuration](../10-user/05-connectors/01-sources/05-http_source.md)

### Dual-Repository Remote Project Mode

Remote project sync now supports a dual-repository mode, allowing:

- `models/`
- `infra/`

to be synchronized from two separate Git repositories.

This makes it easier to maintain rule content and infrastructure configuration independently, which fits more formal team workflows and release processes.

Related docs:

- [Remote project bootstrap and rule hot-reload SOP](../10-user/01-cli/operations/project-sync.md)

### Postgres Source Compatibility Enhancements

Added feature-inheritance support for `Postgres Source` and improved compatibility handling for lower-version scenarios.

This change is mainly intended to make upgrades smoother for existing projects and reduce the amount of connector rework caused by configuration differences.

## Changed

### Error Governance Upgrade

This release upgrades the error-governance stack end to end, with `orion-error` moved to the `0.8` line.

The direct improvements include:

- Error messages now carry more complete path context
- CLI error output is more concentrated and easier to read
- Day-to-day troubleshooting is easier because specific config or rule locations are clearer

## Fixed

### OML `take(...)` Behavior

Synced the upstream fix so that `take(...)` now consumes fields that were already generated in the target record more correctly, and fixed the value-resolution order in same-name field scenarios.

### OML / SQL Parameter Parsing

Synced upstream fixes to improve parsing stability for the following SQL-related cases:

- `group_concat(...)`
- `string_agg(...)`
- `IN (...)`
- `take(field)`
- `__temp_var`

### Runtime Stability

Fixed runtime stability issues in some scenarios.
