# WarpParse 0.22

亲爱的用户们，

我们很高兴地宣布 WarpParse 0.22 版本发布！本次更新主要聚焦于 **运行时可运维性、远端工程同步、知识库能力补强，以及监控与规则体验增强**。如果说 0.20 更强调数据接入能力扩展，那么 0.22 更进一步补齐了“如何稳定上线、如何在线维护、如何观测与回滚”的产品使用闭环。

## 本次更新亮点

0.22 版本的主要变化可以概括为四个方向：

- 运行时管理面更完整：支持标准化的状态查询与重载入口，便于在线实例管理。
- 远端工程同步路径更清晰：从初始化、更新到热重载的 SOP 更统一。
- 知识库与规则能力更实用：知识库配置、SQL/UDF、WPL 路由能力都有更明确的用户文档支撑。
- 监控与可观测性更易落地：配合 VictoriaMetrics、VictoriaLogs 与 Wp-Monitor，可以形成完整观察链路。

## Added

### 新增运行时管理面能力

0.22 补齐了在线运行时最核心的一组运维入口。现在，`wparse daemon` 可以暴露受鉴权保护的 HTTP 管理面，配合 `wproj engine status` 与 `wproj engine reload`，即可完成常见的运行时状态查询与模型重载操作。

当前已明确对外的管理能力包括：

- `GET /admin/v1/runtime/status`
- `POST /admin/v1/reloads/model`
- `wproj engine status`
- `wproj engine reload`

这意味着线上实例不再只能依赖“重启进程 + 人工观察”的方式来变更规则，而是可以围绕统一的 status / reload 入口建立标准化运维流程。

相关文档：

- [运行时管理面使用说明](../10-user/01-cli/operations/admin.md)
- [`wparse` 运行时使用指南](../10-user/01-cli/cli/runtime.md)

### 新增远端工程同步与热更新操作路径

在 0.22 中，围绕远端规则仓库的初始化、更新、重载这条链路，用户文档已经整理成一套可直接落地的 SOP。典型场景包括：

- 首次将远端工程拉取到目标机器
- 后续通过 `wproj conf update` 更新规则工程
- 在不中断 `wparse daemon` 的前提下触发 reload
- 出现问题时按版本执行回滚

对应的日常运维路径已经清晰收敛为：

1. `wproj init --repo <REPO> [--version <VERSION>]`
2. `wproj conf update`
3. `wproj check --what wpl --fail-fast`
4. `wproj engine reload`
5. `wproj engine status`

相关文档：

- [远程工程拉取与规则热更新 SOP](../10-user/01-cli/operations/project-sync.md)
- [`wproj` 项目工具使用指南](../10-user/01-cli/cli/project.md)

### 新增自更新能力

0.22 将二进制自更新能力正式收进 CLI 使用路径。对于通过标准安装方式部署的用户，现在可以直接使用：

```bash
wproj self check
wproj self update --yes
```

这项能力的意义不只是“升级更方便”，更重要的是它把二进制版本切换从手工替换文件，收敛成可检查、可执行、可标准化的运维动作。

相关文档：

- [`wproj` 项目工具使用指南](../10-user/01-cli/cli/project.md)

### 新增知识库配置与 SQL/UDF 使用路径

0.22 对知识库（KnowDB）这条链路的用户侧文档补齐非常关键。现在，知识库能力不再只是“引擎内部特性”，而是有了明确的目录布局、装载规范、安全约束以及最小可运行示例。

当前用户可围绕以下能力使用知识库：

- 用目录式配置初始化权威库
- 使用外置 `create.sql` / `insert.sql`
- 通过 CSV 装载到 SQLite
- 在运行期通过受控表名访问知识库
- 使用内置 UDF 处理 IP 与 CIDR 计算，例如：
  - `ip4_int(...)`
  - `ip4_between(...)`
  - `cidr4_contains(...)`
  - `ip4_text(...)`

这使得 WarpParse 在做富化、归属地匹配、白名单判断等场景时，具备了更实用的“规则 + SQL + 知识库”组合能力。

相关文档：

- [KnowDB 配置](../10-user/02-config/07-knowdb_config.md)
- [OML 集成指南](../10-user/04-oml/05-integration.md)

### 新增监控与链路观察落地方式

0.22 配合监控侧文档，已经能给出一套比较完整的链路观察方案。用户可以基于：

- VictoriaMetrics 存储运行指标
- VictoriaLogs 存储 MISS 数据
- Wp-Monitor 做统一观察入口

来回答几个典型问题：

- 当前链路是否在正常处理数据
- 各个 Source / Parse / Sink 层级是否稳定
- 是否存在明显 MISS 数据
- 下游输出是否出现异常波动

这意味着 WarpParse 的“可观测性”已经不再只是输出几项 metric，而是开始形成一套面向日常巡检与问题排查的使用模式。

相关文档：

- [Wp-Monitor](../10-user/10-monitor/Wp-Monitor.md)
- [Warp Parse 产品概览](../10-user/01-cli/overview/product.md)

### 新增 WPL `json_like` 路由说明

0.22 对 WPL 在“类 JSON 输入处理”上的说明更完整。当前文档已经明确了：

- `json_like` 作为前置轻量 sniff 的定位
- `json` 的内置快速判断能力
- `bad_json` 作为兜底接收原始文本的方式

这对于处理“看起来像 JSON，但不一定是严格 JSON”的日志输入非常关键，可以显著降低规则兜底时的误判成本。

## Changed

### 远端初始化与更新入口进一步统一

0.22 将远端工程同步这条链路的入口表达得更统一了。当前推荐方式是：

```bash
wproj init --repo <REPO> [--version <VERSION>]
wproj conf update
```

如果没有显式指定 `--version`，则优先选择最新 release tag；若远端没有 release tag，则回退到默认分支 `HEAD`。这使得“首次部署”和“后续更新”在操作语义上更加一致。

同时，`wproj conf update`、`wproj engine reload --update` 这些路径也在文档层收敛到了同一套运维叙事里，用户不需要再自己猜“哪个命令该在什么阶段执行”。

## Fixed

### 修复项目初始化时的管理面 token 路径收敛问题

0.22 修复了 `wproj init` 在项目骨架已包含 `[admin_api]` 配置时，token 路径未统一规范的问题。当前更推荐将 token 收敛到项目内的：

```text
runtime/admin_api.token
```

这样做的好处是：

- 避免遗留 `${HOME}/.warp_parse/admin_api.token` 这类用户目录路径
- 让项目目录更自包含，便于部署与迁移
- 降低多项目实例共存时的混淆风险
