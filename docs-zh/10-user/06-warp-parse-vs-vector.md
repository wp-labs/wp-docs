# WarpParse 与 Vector 对比

这篇文档面向正在选型或替换方案的用户，帮助你快速判断：

- 什么时候更适合选择 WarpParse
- 什么时候继续使用 Vector 更合适
- 两者在解析模型、性能和维护方式上的核心差异

## 一句话结论

- 如果你的核心目标是 **高吞吐日志解析、复杂字段提取、规则工程化维护**，WarpParse 更合适。
- 如果你的核心目标是 **通用数据采集、生态连接器覆盖、快速拼装观测链路**，Vector 更合适。

两者都能做日志处理，但产品重心不同：

- **WarpParse** 更偏向“高性能解析引擎 + 规则工程体系”
- **Vector** 更偏向“通用可观测性数据管道”

## 核心定位对比

| 维度 | WarpParse | Vector |
| --- | --- | --- |
| 产品定位 | 面向日志解析与安全数据处理的 ETL 引擎 | 面向日志、指标、追踪的通用数据管道 |
| 强项 | 高性能解析、复杂文本结构提取、规则治理 | 连接器生态、通用采集转发、社区成熟度 |
| 主要建模方式 | WPL + OML | TOML + VRL / 内建 transform |
| 典型使用场景 | 安全日志标准化、复杂异构日志解析、解析规则平台化 | 统一采集、路由转发、可观测性链路接入 |
| 更关注的问题 | 解析精度、规则表达力、吞吐与延迟 | 数据接入广度、部署便捷性、生态兼容性 |
| 国产信创支持 | 更有针对性，兼容性更强 | 不是主要优势方向 |

## 解析模型差异

### WarpParse

WarpParse 把处理过程拆成两个主要阶段：

- **WPL**：负责原始日志解析与字段提取
- **OML**：负责对象组装、字段映射、转换与输出模型整理

这种拆分的好处是：

- 解析规则和对象建模职责分离
- 复杂日志的可维护性更好
- 团队可以按“解析规则”和“输出模型”分工协作

### Vector

- VRL 或内建 transform 负责字段提取、变换、路由

## 规则维护方式对比

| 维度 | WarpParse | Vector |
| --- | --- | --- |
| 解析与转换职责 | 拆分明确 | 常在同一条处理链上完成 |
| 复杂文本日志 | 更适合 | 可以做，但规则容易膨胀 |
| 规则复用 | 更适合做规则工程化管理 |  |
| 调试关注点 | 命中哪条 WPL、如何映射 OML |  |

如果你的场景里有下面这些特征，WarpParse 的优势会更明显：

- 日志格式很多，而且经常变化
- 需要从非结构化文本中稳定提取大量字段
- 需要把解析规则交给专门团队长期维护
- 需要同时兼顾吞吐、精度和规则可测试性

如果你的场景更接近下面这些情况，Vector 往往更顺手：

- 主要目标是采集并转发，而不是深度解析
- 已经大量使用 Vector 生态或周边模板
- 日志格式相对稳定，解析逻辑不复杂
- 需要快速接入多种 source / sink

## 性能差异

从当前仓库内的基准测试结果看，WarpParse 在“解析为主”的场景下，整体吞吐通常明显高于 Vector，尤其在以下场景更明显：

- 小包高并发日志
- 大字段、长文本日志
- 解析后还需要继续做字段转换
- TCP 输入到文件输出这类端到端路径

例如在 `Mac M4 Mini` 基准中：

- Nginx `File -> BlackHole`，WarpParse 为 `2,789,800 EPS`，Vector-VRL 为 `572,076 EPS`
- APT `File -> BlackHole`，WarpParse 为 `328,000 EPS`，Vector-VRL 为 `37,777 EPS`

详细数据可参考：

- [BenchMark Report](../20-report/benchmark.md)
- [性能基准测试报告](../20-report/report_mac.md)

不建议把一组基准数字直接外推到所有生产环境，实际结果仍会受日志结构、I/O 拓扑、规则复杂度和下游 sink 影响。

## 生态与接入方式对比

| 维度 | WarpParse | Vector |
| --- | --- | --- |
| Source / Sink 覆盖 | 当前Source `5` 个、Sink `15` 个 | 更广，更成熟 |
| 社区生态 | 偏产品与项目工程实践 | 开源生态更大 |
| 可观测性场景 | 可承接，但不是唯一重心 | 覆盖日志/指标/追踪 |
| 安全日志工程 | 更强 | 可以做，但通常不是最佳表达方式 |
| 国产信创兼容性 | 更强，适合国产化环境落地 | 不是主要卖点 |

按当前中文文档入口统计，WarpParse 已文档化的连接器数量为：

- `Source 5` 个：`file`、`kafka`、`syslog`、`http`、`tcp`
- `Sink 15` 个：`file`、`syslog`、`prometheus`、`tcp`、`victorialogs`、`doris`、`kafka`、`mysql`、`arrow-ipc`、`elasticsearch`、`arrow-file`、`clickhouse`、`http`、`postgresql`、`dmdb`

这说明 WarpParse 在社区版里并不只是“解析引擎”，而是已经具备一套可独立落地的连接器能力。

## 国产信创支持

对很多政企和安全场景来说，是否能更顺利地落地在国产化基础设施上，是一个单独的重要维度。

在这一点上，WarpParse 的优势可以单独拎出来看：

- 对国产信创环境支持更有针对性
- 在国产化部署场景下兼容性更强
- 更适合作为需要兼顾性能、解析能力和国产化适配的方案

如果你的项目明确要求：

- 国产操作系统兼容
- 国产软硬件环境适配
- 信创场景下的稳定部署与交付

那么这会是 WarpParse 相比 Vector 的一个明确优势。

## 社区版完整度对比

这里的“完整度”主要是指：用户拿到社区版后，能否较完整地完成下载、阅读、学习、试用、规则编写、运行观察等工作。

| 维度 | WarpParse 社区版 | Vector 社区版 |
| --- | --- | --- |
| 程序 | 可直接运行核心解析链路 | 可直接运行完整数据管道 |
| 文档 | 文档体系已经较完整，覆盖产品、规则、配置、连接器、教程与基准报告 | 文档成熟，覆盖通用 pipeline 与生态集成 |
| 示例 | 有专门的 `wp-example` 项目，下载后即可运行 | 分散在官方文档、仓库和社区内容中 |
| 规则编辑器 | 提供专门的 `wp-editor` | 无 |
| 处理监视器 | 提供专门的 `wp-monitor` | 第三方 |
| 兼容性 | 主流系统平台+国产信创，兼容性更强 | 主流系统平台 |

如果从社区版完整度来判断，WarpParse 的优势更直接：

- 有完整程序可运行
- 有成体系文档可阅读
- 有 `wp-example` 可直接下载试用
- 有 `wp-editor` 支撑规则开发
- 有 `wp-monitor` 支撑处理监视与观察
- 有更强的国产信创兼容性

因此在“拿到社区版后，是否能独立完成学习、试用、规则开发和运行观察”这个维度上：

- **WarpParse 的完整度明显优于 Vector**

## 常见判断

### 我已经在用 Vector，是否需要迁移到 WarpParse？

如果你当前主要痛点是：

- VRL 规则越来越长
- 正则难维护
- 解析吞吐不足
- 安全日志标准化成本高

那么值得评估迁移一部分“解析密集型链路”到 WarpParse，而不是一开始就全量替换。
